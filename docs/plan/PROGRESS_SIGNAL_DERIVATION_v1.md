# Progress Signal Derivation v1

Status: Deferred product canon  
Purpose: Record how future progress, mastery, and retention signals must derive from real learning/scenario truth instead of fake gamification.  
Use: Reference this document when future Sharky, streak, badge, profile, or mastery work needs to decide what counts as meaningful progress.

## Why This Document Exists

The project now has enough real learning structure to support honest progress later:
- source-driven scenarios
- engine-native table-state
- skill-family thinking
- multi-step and capstone scenarios
- headless validation for authored scenario truth

That means future progression features do not need to guess.
They can derive their signals from real learning evidence.

This document exists to keep future progression grounded in:
- scenario truth
- repeated performance
- skill-family development
- meaningful completion

and to block fake progress systems that reward noise instead of learning.

## Core Principle

Progress signals must amplify learning truth, not simulate it.

That means:
- signals should come from authored scenario/data truth
- signals should reflect repeated evidence, not one lucky event
- signals should track real skill-family growth, not generic activity only
- completion, confidence, and mastery should remain distinct ideas

## What Future Progress Signals Should Derive From

Allowed future progress signals should come from real learning evidence such as:

- scenario correctness
- repeated stability inside a skill family
- capstone or checkpoint completion
- linked scenario-chain performance
- recovery after earlier weakness
- consistency across related scenario types

The underlying source should remain authored scenario/data truth plus deterministic evaluation outcomes.

## Allowed Signal Sources

### Scenario Correctness

A correct answer on a real scenario is a valid signal.

What it can mean:
- the learner solved one real authored spot correctly

What it does not mean by itself:
- stable understanding
- mastery
- transfer ability

Single-scenario correctness should be treated as signal, but not strong signal.

### Repeated Stability On A Skill Family

This is one of the strongest future signal types.

Examples:
- repeated correct position/initiative judgments
- repeated correct hand-comparison answers
- repeated correct texture/outs intuition answers
- repeated simple action follow-ups grounded in the same family

This is stronger than isolated correctness because it shows pattern recognition instead of one-off luck.

### Capstone / Checkpoint Completion

Capstones and checkpoints should carry more weight than isolated drills because they:
- connect multiple ideas
- require cumulative understanding
- better reflect linked table reasoning

They are not the only signal, but they should be premium signal.

### Linked Scenario-Chain Performance

Multi-step chains are stronger evidence than isolated steps when they stay on one coherent scenario and require the learner to connect ideas in sequence.

Examples:
- position -> initiative -> action
- texture -> outs -> continue/fold intuition
- capstone chains that connect multiple World 2 ideas

This should be treated as stronger than flat drill count alone.

### Recovery / Improvement After Earlier Weakness

A future progression system should give credit for real improvement, not only flat success.

Examples:
- a skill family that was unstable but later becomes consistent
- a checkpoint failed earlier but passed later
- repeated mistake pattern shrinks over time

Improvement is a real signal because it reflects learning movement, not just raw score.

## Weak Or Unsafe Signals

These should be rejected or heavily down-weighted if used later:

### Raw Clicks / Taps

Clicks only prove interaction happened.
They do not prove learning.

### Session Count Alone

Finishing many sessions can indicate persistence, but not understanding.
Session count can support progress context, but not carry mastery on its own.

### Time Spent Alone

Time spent can reflect effort, confusion, or distraction.
It is not a reliable learning signal by itself.

### Isolated Lucky Success

One correct answer is not stable evidence.
It should not create large mastery jumps.

### Streak Alone

A streak can support return behavior, but it does not prove skill growth.
Streak should never be treated as equivalent to competence.

### Completion Alone

Completion and understanding are not the same thing.
Completing a session without strong correctness or later stability should not be inflated into mastery.

## Skill-Family Grounding

Future progress should map to real learning families already reflected in project canon.

Examples of current/future grounding:

- hand reading / comparison
- position / initiative
- board pressure / texture
- improvement / outs intuition
- simple action follow-up
- later preflop framework / hand categories
- later value / bluff / sizing purpose
- later equity / pot-odds intuition

Progress should not be stored as one undifferentiated “good at poker” number.
It should remain connected to skill-family evidence.

## Signal Quality Rules

### Repeated Evidence Beats Single Events

A learner should not look “mastered” from one result.
Confidence should grow with repeated coherent evidence.

### Capstones / Checkpoints Carry More Weight

A capstone or checkpoint should count more than a trivial single-step drill because it reflects connected understanding.

### Completion And Correctness Are Different

Completing something matters.
Getting it right matters more.
Getting it right repeatedly matters more still.

### Stable Cross-Context Performance Is Stronger

If a learner performs well across multiple scenario shapes within the same skill family, that is a stronger progress signal than success on one narrow format only.

### Recovery Matters

Progress systems should recognize improvement after earlier weakness.
The goal is not only to reward strong learners, but to make real learning movement visible.

## What Should Not Count As Meaningful Progress

The following should never be treated as high-quality progress by themselves:
- opening the app
- tapping through content quickly
- raw attempt count
- pure exposure without outcome
- a mascot reaction without learning evidence
- completion inflation disconnected from scenario truth

If a signal can be farmed without learning, it is a bad signal.

## Near-Term Implication

Near-term implication:
- keep scenario/data work compatible with future progress derivation
- keep skill-family boundaries legible
- keep capstones/checkpoints meaningful
- do not ship fake progression before these grounded signals exist

The current learning/scenario foundation should continue to be treated as the primary future source of honest progress.

## Decision Rule

If a future progression feature proposal cannot explain:
- which real scenario truths it derives from
- which skill family it reflects
- why the signal means learning instead of noise

then it is not ready.
