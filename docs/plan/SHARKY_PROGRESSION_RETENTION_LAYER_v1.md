# Sharky / Progression & Retention Layer v1

Status: Deferred product canon  
Purpose: Record the intended future retention/progression layer so it can be implemented later in the right order on top of the proven learning and scenario foundation.  
Use: Reference this document when deciding what the future “soul of the app” layer should do, what should come early versus later, and what should be rejected as fake or premature gamification.

## Why This Layer Exists

The learning core now has enough real substance to support a stronger retention/product layer later:
- source-driven scenarios
- engine-native table-state
- multi-step/capstone scenarios
- headless validation for authored scenario truth
- early intuition bridges that feel usable, not just theoretical

That means the next major multiplier is no longer “more raw structure only.”  
It is the layer that makes progress feel alive and worth returning to:
- emotional continuity
- visible progress
- identity
- milestone memory
- long-term return loops

This layer should make the app feel more like a living coach-product, not a sterile drill bank.

## Core Principle

Gamification must amplify learning truth, not replace it.

That means:
- real scenario performance should drive progress signals
- real skill-family growth should drive mastery signals
- real completion/capstone evidence should drive celebration
- the retention layer should reinforce useful behavior, not fabricate importance

This layer must never become a substitute for good teaching.

## What This Layer Is For

This future layer should:
- give the product a soul and identity
- make progress feel visible and cumulative
- strengthen return loops without breaking the table-first experience
- support long-term retention and later monetization only after real value is felt
- turn the scenario/data foundation into meaningful learner-facing progress signals

## Early High-EV Layer

This is the first stage worth shipping after the learning core is strong enough.

### Sharky As Compact Coach

Sharky should be:
- a guide
- a tone-setter
- a reinforcement layer

Sharky should not be:
- an open-ended chat system
- an infinite content generator
- a noisy interruption engine

Early Sharky scope:
- one curated phrase before a session
- one short reaction after success or error
- one identity reinforcement line after meaningful progress

Governed early phrase-pack categories:
- pre-session encouragement
- post-correct reaction
- post-mistake repair tone
- progress reinforcement
- milestone celebration

### Streak-Lite

Use a lightweight streak system only if it reinforces return behavior without creating punishment.

Good early streak behavior:
- simple daily return signal
- low-pressure continuity
- visible but not dominant

Avoid:
- guilt-heavy reset framing
- punishing loss of long streaks
- friction that overwhelms learning value
- using streak loss or reward withdrawal as emotional pressure

### Mastery / Progress By Skill Family

The app should show progress by real learning families, not only by generic “levels.”

Early high-EV examples:
- hand reading / comparison
- position / initiative
- board pressure / texture
- improvement / outs intuition
- simple action follow-up

This should come from actual scenario performance and coverage, not generic completion counts alone.

### Milestone / Capstone Celebration

When the learner completes a meaningful block or capstone:
- celebrate it briefly
- make the progress legible
- reinforce what they can now do

The celebration should feel earned, compact, and tied to real learning.

Allowed early earned celebration events:
- repaired weak pattern
- completed world block
- meaningful skill-family gain
- daily return without pressure

### Meaningful Badges

Early badges should exist only for real milestones:
- completing a block
- stabilizing a skill family
- finishing a capstone/checkpoint

Badges should indicate signal, not noise.

### Profile Summary / Learning Identity

The early profile layer should answer:
- what the learner is getting better at
- what they have completed
- what kind of player identity is emerging

This should feel like a learning identity, not a stat junkyard.

## Mid-Layer

After the early layer proves useful, the next stage can add richer but still grounded meta structure.

Examples:
- a stronger profile page
- rank/title layer
- Sharky recap summaries
- deeper per-skill progress summaries
- clearer world/block completion history

This layer should still stay grounded in real scenario/skill truth.

## Later / Deeper Layer

Only after the learning core and early retention layer are clearly working should deeper meta systems be considered.

Possible later features:
- mastery levels
- trait-style progress identities
- RPG-style progression depth
- richer long-term meta loops
- economy hooks only if the product already has clear durable value

This later layer should deepen motivation, not substitute for weak learning value.

## Anti-Patterns

Avoid:
- fake XP for meaningless clicks
- noisy mascot behavior
- infinite AI-coach chatter
- dashboard spam
- gamification before learning truth
- hard friction too early
- monetization pressure before felt value
- anything that disrupts the table-first experience

If a retention feature creates noise without improving return motivation or learning confidence, it is the wrong feature.

## Sequencing Rule

Rollout order should stay explicit:

1. learning core first  
2. visible progress + compact Sharky coach layer  
3. profile identity + stronger progression summaries  
4. deeper mastery / RPG / meta systems later

The product should not skip straight to deep meta systems before the learner already feels real progress and useful teaching value.

## What Must Stay Primary

These remain higher priority than the retention layer:
- learning truth
- table-first UX
- deterministic scenario quality
- skill-family coherence
- real authored capstones/checkpoints

The retention layer should sit on top of these, not compete with them.

## Practical Near-Term Implication

This layer is deferred on purpose.

Near-term implication:
- continue strengthening the scenario/data-driven learning core
- keep progress signals derivable from real scenario truth
- do not implement Sharky/progression opportunistically in scattered places
- when this layer starts, begin with the compact early high-EV set only

## Decision Rule

If a future feature proposal for this layer does not:
- reinforce real learning truth
- strengthen return motivation cleanly
- preserve table-first UX
- fit the sequencing rule above

then it should not ship yet.
