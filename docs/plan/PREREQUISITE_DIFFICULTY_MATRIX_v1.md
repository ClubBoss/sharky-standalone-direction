# Prerequisite Difficulty Matrix v1

Purpose:

- define the first canonical prerequisite and difficulty truth layer for future world / module / session expansion
- make unlock order and dependency expectations explicit instead of relying on local intuition
- stay bounded to prerequisite / difficulty truth only, not release gating, boredom control, monetization, or runtime implementation

## Governs Now

This SSOT governs:

- the major foundation skill groups that should precede later skills
- what cannot unlock before what
- where soft branching is allowed
- where mastery gates are required
- how high-level difficulty tiers should be interpreted

This SSOT does not yet govern:

- runtime unlock logic
- release gate policy
- content factory readiness
- anti-boredom / novelty systems
- pricing, entitlement, or monetization

## Canonical Skill Group Order

Use this as the default dependency ladder:

1. table orientation and locate anchors
2. order / sequence basics
3. single-step classification and simple recognition
4. compact action choice
5. compact size / purpose choice
6. visible comparison / showdown / counting families
7. short multi-step reasoning
8. broader mixed application and transfer

Interpretation:

- later groups may reference earlier ones
- earlier groups should not depend on later groups to be understandable

## Major Skill Groups

| Group | Examples | Must precede |
| --- | --- | --- |
| orientation anchors | seat, button, blind, street, board anchors | all later decision and comparison work |
| sequence / turn order | action order, blind order, street order | initiative, action choice, multi-step chain |
| recognition / classifier basics | board texture, simple category labels, obvious draw/hand recognition | action choice, outs/equity intuition, mixed comparison |
| compact action choice | fold/call/check/bet/raise decisions with local reasons | sizing purpose, pressure logic, mixed application |
| size / purpose choice | meaningful size alternatives and local purpose | higher-pressure strategic application |
| visible comparison / counting | showdown comparison, outs, simple odds intuition | mixed application, transfer, later math-heavy work |
| short chain reasoning | 2-4 linked steps with stable semantics | capstones, mixed challenge, transfer |
| transfer / mixed application | blended stable families across contexts | later world integration, mastery loops |

## Hard Dependency Rules

These are non-optional order rules for future expansion:

- orientation anchors must come before any task that assumes stable spatial/table reference
- sequence basics must come before initiative or action-order-dependent choice
- recognition / classifier basics must come before applied decision families that depend on those labels
- compact action choice must exist before size / purpose choice is treated as meaningful strategy rather than decoration
- single-family stable understanding must exist before short multi-step chaining
- short multi-step chaining must exist before broad mixed challenge or transfer

## Soft Branching Rules

Soft branching is allowed when:

- two candidate skills share the same prerequisite base
- neither branch introduces a stronger dependency than the other
- the branch does not skip the current world’s required foundation group

Examples of allowed soft branching:

- after basic recognition, branch into visible showdown comparison or outs counting
- after compact action choice, branch into bet sizing choice or bounded policy repetition

Soft branching is not allowed when it would:

- bypass sequence/order understanding
- bypass the first stable decision family
- mix advanced comparison/math with unstable recognition basics

## Mastery Gate Rules

Mastery gates are required at these boundaries:

- before moving from recognition/classifier work into applied decision work
- before moving from isolated single-step work into short chain reasoning
- before moving from single-family understanding into mixed review/application
- before introducing transfer-to-live-table style work

Mastery gates are not required:

- between two examples of the same stable concept
- between small difficulty rises inside the same family when no new dependency is crossed

## Prerequisite vs Pacing

Use this distinction:

- prerequisite:
  - a learner cannot reasonably understand or perform the later skill without the earlier one
- pacing:
  - the learner could understand the next step, but the rhythm should still slow down for retention, recap, or density control

Practical rule:

- if skipping the earlier skill makes the later one conceptually confusing, it is a prerequisite
- if skipping the earlier skill mainly makes the experience too abrupt or dense, it is pacing

## Difficulty Tier Meaning

Use high-level tiers only:

| Tier | Meaning |
| --- | --- |
| Tier A | anchor / recognition / simple locate work with low ambiguity |
| Tier B | basic classification and simple comparisons with explicit visible truth |
| Tier C | compact decision choice with one clear local reason |
| Tier D | bounded purpose tradeoff or short linked reasoning |
| Tier E | mixed application, checkpoint, or transfer requiring stable earlier tiers |

Interpretation rules:

- tier rise should usually follow the pacing SSOT one step at a time
- a later tier may still revisit earlier concepts, but should not assume missing lower-tier foundations
- difficulty tier is not the same as content importance; foundational content may be low-tier but mandatory

## Unlock Matrix

| Later skill area | Minimum prerequisite base |
| --- | --- |
| board texture / simple classifiers | orientation anchors + basic visible recognition |
| position / initiative reasoning | orientation anchors + sequence basics |
| action choice | recognition/classifier basics + sequence basics |
| sizing purpose | compact action choice |
| visible showdown comparison | recognition/classifier basics |
| outs counting | visible comparison basics + recognition of draw structures |
| pot-odds / continue-price intuition | outs/counting or equivalent simple improvement intuition + compact action choice |
| short hand chains | stable single-step semantics in each included family |
| mixed challenge / capstone | at least 2 already-stable families + prior checkpoint/recap support |
| transfer / real-table application | stable mixed challenge capability |

## Anti-Jump Preservation Rules

- do not unlock a later family simply because the UI can render it
- do not treat host reuse as proof of curricular readiness
- do not count adjacent vocabulary as a prerequisite if the cognitive job is different
- do not collapse “seen once” into “mastered enough to chain”

## Strategic Use Rule

Before adding a new world/module/session slice, ask:

1. Which skill group does it belong to?
2. What explicit prerequisite base does it assume?
3. Is that base already stable in the intended path?
4. Does it require a mastery gate or only pacing discipline?
5. Is the claimed difficulty tier consistent with the real dependency jump?

If those answers are unclear, the slice is not ready for default path expansion.
