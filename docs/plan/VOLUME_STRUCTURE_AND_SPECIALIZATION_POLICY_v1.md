# Volume Structure And Specialization Policy v1

Status: ACTIVE
Last updated: 2026-05-06

## Purpose

Define the canonical high-level route shape for the full course so future
content production does not drift between:

1. one endless 36-world blob
2. too-early format branching
3. unclear specialization timing

This document is a route-policy owner.
It does not replace per-world planning.

## Core Decision

The product should be structured as:

1. one shared core route first
2. one specialization gateway after the core
3. later format-specific mastery after the gateway

That means:

1. no hard Cash vs MTT fork during the beginner shared spine
2. no requirement that a novice choose a permanent format identity too early
3. real specialization later, when the learner has earned the context needed
   to choose well

## Three-Volume Structure

### Volume I. Core Foundation

Worlds: `W1-W12`

Job:

- take a true beginner from zero to independently competent early play
- teach durable anchors that survive across formats
- build enough confidence and transfer that the learner can choose a direction
  intelligently later

Release role:

- this is the launch-critical route
- this is the first independent wave
- this should be marketable as a complete beginner-to-strong-foundation product

### Volume II. Applied Postflop And Strategic Depth

Worlds: `W13-W24`

Job:

- deepen postflop logic, range reasoning, and real strategic process
- convert the learner from "safe and competent" into "strong and practical"
- prepare the learner for honest format specialization

Release role:

- later expansion after Volume I is stable
- may remain locked / coming soon during the first launch phase

### Volume III. Specialization And Pro Layer

Worlds: `W25-W36`

Job:

- deliver format-specific mastery and later professional workflow
- support deeper Cash / MTT / SNG / live / full-ring / advanced-study paths
- complete the route from novice to highly dangerous specialist

Release role:

- advanced expansion, not first-wave obligation

## Specialization Timing Rule

Cash and MTT should not become separate beginner products.

Canonical timing:

1. `W1-W12`: one shared spine
2. `post-W12 gateway`: first explicit specialization recommendation
3. `W13-W24`: deeper strategic worlds may begin tagged or variant treatment
4. `W25-W36`: full specialization is allowed

## Why The Split Comes After W12

Before W12, the learner still needs one stable grammar for:

1. table literacy
2. hand discipline
3. position
4. preflop structure
5. bet purpose and price
6. board and draw reading
7. range basics
8. stack/risk awareness
9. pressure awareness
10. real-play transfer
11. process-over-outcome mindset

Splitting earlier would:

1. double content cost too soon
2. create false precision before the learner understands the tradeoffs
3. weaken the beginner spine
4. make launch slower and messier

## Specialization Gateway Contract

After Volume I, the learner may enter a specialization gateway.

This is not a random preference picker.
It should combine:

1. stated preference
2. play environment
3. session-length preference
4. variance tolerance
5. pressure comfort
6. short scenario-based diagnostic

Gateway outputs should be:

1. `Recommended: Cash`
2. `Recommended: Tournament`
3. `Not sure yet / Mixed bridge`

## Branching Model

Use this branching order:

1. shared concept first
2. format-tagged examples next
3. true branch only when the concept meaning changes materially by format

Meaning:

1. do not fork because labels differ
2. fork when incentives, thresholds, or survival logic differ enough that one
   route would teach the wrong lesson

## Product Packaging Rule

The three-volume structure is not only for internal planning.
It is also the preferred user-facing packaging model.

Recommended framing:

1. Volume I: available now
2. Volume II: planned / coming soon
3. Volume III: advanced / specialization / coming later

This keeps the roadmap honest without making the product feel unfinished.

## Hard Rules

1. Do not hard-fork Cash vs MTT before the shared spine is complete.
2. Do not market W13-W36 as equally complete if Volume I is the active launch
   route.
3. Do not create learner-visible format branches just because old main had the
   taxonomy.
4. Do create a real specialization gateway after Volume I is proven stable.
5. Do preserve later full specialization as part of the long-term route, not as
   an optional afterthought.

## Current Practical Recommendation

For current execution:

1. treat `W1-W12` as the canonical launch route
2. keep `W13+` locked / coming soon unless seam and density proof exists
3. preserve future Cash / MTT specialization in docs and runtime seams
4. do not open full branch production before the `W12 -> W13` frontier is
   honestly ready
