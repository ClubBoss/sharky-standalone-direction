# Value / Monetization Packaging Planning v1

Status: planning/audit only
Date: 2026-06-18

## 1. Executive verdict

Sharky is ready to start monetization and value-packaging planning.

Sharky is not ready for public paywall implementation, public subscription
purchase, or public Premium Hub exposure.

The accepted first-week proof loop now clears the product-trust precondition:
the learner can see a table clue, make a decision, receive signal-bound
feedback, complete a small daily loop, and see concrete Review/Profile proof
without being pressured to pay. That is enough to begin a soft premium value
preview.

The next commercial step should not be a paywall. It should be a low-pressure
premium value preview after useful learning proof, using only claims that map to
current deterministic product truth.

## 4. Trust-safe monetization principles

1. Value before friction.
   - Placement, first hand, first feedback, and the first daily loop stay free.

2. Premium sells depth, not permission to begin.
   - The learner should feel, "I can learn here for free, and premium gives me
     more focused practice."

3. Premium claims must be evidence-bound.
   - Copy can reference table clues, extra reps, repair depth, route depth, and
     review sharpening only where the app already proves the underlying concept.

4. No fake personalization.
   - Use deterministic language: "based on the clue you just practiced" or "more
     reps on this table read." Do not say AI, adaptive AI, solver, GTO, automatic
     leak fixing, or guaranteed improvement.

5. Commerce truth must lag value packaging until safe.
   - A preview can explain later value. It must not create a real purchase path
     while public receipt, restore, refund, cancellation, and entitlement truth
     are incomplete.

6. Upgrade should feel optional and earned.
   - The best premium moment comes after the app has helped once, not before the
     user knows whether Sharky works.

## 5. Recommended first premium touchpoint

Recommended first touchpoint:

**After the first completed daily/session result, surfaced as a passive premium
value preview from Home or Practice completion.**

Why this is the right first moment:

- The learner has already seen table-signal feedback.
- The learner has completed a short daily loop.
- The result ceremony has a concrete improvement anchor.
- The touchpoint can be optional and non-blocking.
- It avoids interrupting the strongest first-value moment.

Do not show premium:

- before placement starts;
- during placement questions;
- on placement result before the first useful hand;
- immediately on first correct feedback;
- as a hard gate in Review repair;
- as a modal that blocks the daily completion reward.

Secondary future touchpoint:

- after a completed repair, where premium can be framed as "more repair reps for
  this kind of missed clue." This should come after the daily-completion preview
  because repair can be emotionally sensitive.

## 6. Free vs premium boundary

### Free should include

- placement and beginner-safe route calibration;
- first useful hand;
- first correct/wrong feedback proof;
- same-signal next rep;
- first daily trainer loop;
- basic Home return reason;
- basic Review repair item and direct fix;
- enough early route access to establish trust, including the public free
  foundation boundary defined by active product policy;
- honest progress receipts and Profile rhythm.

Free must prove:

- Sharky teaches visible table reads;
- mistakes become useful repairs;
- returning tomorrow has a concrete reason;
- premium is not needed to understand the product.

### Premium should eventually own

- more reps on table clues the learner has already seen;
- deeper same-signal practice sets;
- richer repair resurfacing and weak-spot depth;
- longer route depth beyond the free foundation;
- mastery replay and stronger long-horizon sharpening;
- later specialization and advanced route depth when those systems are ready.

### Do not sell as premium yet

- AI coaching;
- solver/GTO training;
- guaranteed win-rate improvement;
- personal leak profile automation;
- all worlds/features unlocked;
- seven-day plan, progress insights, or review queues as paid benefits unless
  exact entitlement mapping and tests prove them;
- restore/cancel/refund guarantees before store-policy copy and receipt truth
  are production-ready.

## 7. Honest premium promise

Allowed promise shape:

- "More practice on the table clues you are already learning."
- "Extra reps after the free foundation."
- "Deeper review of missed reads."
- "Longer route depth when you want more than the opening path."
- "Premium stays optional; the first useful loop stays free."

Best commercial proposition:

**Free proves Sharky can improve your table read. Premium gives you more reps,
deeper repair, and a longer path once you want to sharpen faster.**

Deferred promise shape:

- "Personal leak profile" waits for a real skill/leak profile system.
- "Adaptive plan" waits for deterministic personalization proof or a separately
  honest rule-based system.
- "Full route unlock" waits for explicit entitlement mapping and launch policy.
- "Subscription restore and account access" waits for production receipt and
  restore convergence.

## 8. Banned claims / banned patterns

Banned wording:

- "AI coach"
- "AI-personalized"
- "GTO"
- "solver"
- "optimal frequencies"
- "guaranteed improvement"
- "guaranteed win rate"
- "fix leaks automatically"
- "all features unlocked"
- "restore always works"
- "cancel anytime" until store subscription-management copy is finalized
- "offline/private/local-only" until privacy/data posture is audited
- "seven-day guided plan" as a premium promise until entitlement and content
  truth support it

Banned patterns:

- hard paywall before first learning value;
- paywall immediately after first feedback;
- trial pressure before daily completion or repair proof;
- fake urgency, fake discount countdowns, or guilt-based streak pressure;
- blocked basics framed as premium depth;
- premium modal louder than the result ceremony;
- Runout-style commercial pressure without Sharky's proof-first trust layer.

## 9. Suggested premium preview UX

Recommended surface:

- a small, optional preview entry on the completed Practice/Home return state;
- no purchase CTA;
- no price;
- no trial start;
- no store restore;
- primary action remains the learning action or free-route continuation;
- preview action may be a quiet secondary CTA such as "See what premium adds";
- preview sheet keeps "Stay on free route" as the dominant trust action.

Suggested hierarchy:

1. You improved a real table read.
2. Free keeps the opening path useful.
3. Premium later adds more practice on this kind of clue.
4. Stay on free route.

Recommended content shape:

- one proof line: the clue or skill the learner just warmed;
- one free boundary line: first path/daily practice remains free;
- two or three premium value points, each tied to current truth;
- one quiet "not now" or "stay free" close action.

Avoid:

- pricing cards;
- countdowns;
- annual/monthly plan selection;
- App Store restore language;
- modal auto-open after first feedback;
- any premium message that obscures the table/session completion proof.

## 10. Paywall timing recommendation

Do not implement paywall now.

Allowed now:

- preview-only premium value packaging after first-week proof;
- copy guardrails;
- claim truth tests if a later implementation touches copy;
- design of a future paywall gate.

Paywall may be considered only after:

- entitlement ledger is the single read/write authority;
- public purchase path no longer uses mock purchase behavior;
- receipt verification policy is production-grade;
- restore converges into the same entitlement ledger;
- trial, expiry, refund, cancellation, and revocation states have user-safe copy;
- the premium/free route boundary is explicit and tested in active Act0;
- App Store / Play Store subscription policy copy is ready;
- first-value and first-return flows remain unblocked by monetization.

Recommended timing:

1. first session: no monetization;
2. first feedback: no monetization;
3. first daily/session completion: passive preview allowed;
4. first repair completion or later return: optional preview allowed;
5. route boundary after free foundation: future paywall candidate only after
   commerce truth is production-ready.

## 11. Technical commerce readiness checklist

Before public commerce or paywall:

- one entitlement ledger owns premium/trial/purchase/restore status;
- legacy premium flag and purchased-product cache cannot independently grant
  production access;
- public upgrade does not call a mock gateway path;
- App Store / Play Store product IDs are final and mapped explicitly to
  entitlement records;
- receipt verification is not a local always-true stub;
- restore writes or refreshes the same ledger used by active UI gates;
- subscription expiry, refund, revocation, grace period, and cancellation states
  have deterministic behavior;
- trial status uses the same access pipeline as paid premium;
- public copy for restore, cancellation, refunds, support, privacy, and terms is
  approved;
- tests prove paid/restore/trial/free transitions through active Act0 gates;
- Premium Hub remains hidden until all above are true.

## 12. Runout comparison

Runout remains stronger in visible commercial funnel maturity:

- aggressive onboarding segmentation;
- subscription packaging;
- paywall ceremony;
- promotional offer framing;
- polished commercial rhythm.

Sharky's advantage is different and should be protected:

- value before monetization;
- beginner-safe table-first learning;
- deterministic signal proof;
- repair proof;
- same-signal continuation;
- no fake AI or solver promise.

What to borrow as principle:

- clear value packaging;
- calm premium hierarchy;
- concise benefit framing;
- confidence that there is a deeper product behind the free path.

What not to copy:

- early hard paywall;
- unsupported rating/performance claims;
- AI/adaptive language without technical proof;
- GTO-heavy intimidation for beginners;
- discount-first pressure before trust.
