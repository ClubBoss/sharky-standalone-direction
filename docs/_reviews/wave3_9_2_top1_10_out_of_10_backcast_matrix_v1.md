## 6. Per-Wave Definition Of Done

For Waves 3.10-3.15, every Retro Summary must include:

- target 10/10 block;
- current gap;
- learner-visible change shipped;
- exact files changed;
- tests/checks run;
- screenshot/device proof;
- expected TOP1 movement;
- actual observed movement or reason it cannot be measured yet;
- anti-theater proof;
- what was explicitly not built;
- next wave validity.

## 7. Acceptance Thresholds

Wave success is not "no blocker."

Wave success is:

`the named 10/10 block moved`

If the block does not move, the wave must return one of:

- `blocked_missing_prerequisite`
- `implemented_but_no_visible_movement`
- `dormant_hypothesis`
- `needs_follow_up_p1`

Only `blocked_missing_prerequisite` may insert a prerequisite wave, and only if
the prerequisite cannot fit inside the current wave.

## 8. Guard vs Excellence Reinforcement

Guard waves protect route truth, claim safety, regressions, and blockers.

Excellence waves must improve the learner's experience in the named 10/10
block. A clean audit can inform an excellence wave, but it does not complete
one unless the wave was explicitly docs-only route/SSOT control.
