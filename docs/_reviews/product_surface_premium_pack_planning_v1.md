## 3. Executive Verdict

Sharky is visually good enough to continue product work. It does not need a broad redesign now.

The active proof loop is stronger than the packaging layer. Play/feedback is already the commercial differentiator: it proves "choice -> table signal -> why -> next useful rep." Home, Practice, Review, Learn, and Profile are no longer blocker-level, but they still do not share a consistent premium rhythm. The remaining issue is a cross-surface commercial floor, not one broken screen.

Single next highest-EV implementation wave: **Product Surface Premium Pack v1**.

That wave should be a bounded surface grammar pass across the active first-week non-table surfaces. It should not touch table geometry, learning content, route order, telemetry, commerce, screenshot tooling, or localization. Its job is to make the first-week app feel like one calm premium trainer instead of a set of individually improved cards.

## 5. Top Commercial Gaps

1. Cross-surface premium rhythm is still uneven. Home, Practice, Review, Profile, and result states are individually adequate, but they do not yet share one obvious commercial grammar for header, proof, progress, and CTA hierarchy.

2. Session/result ceremony is truthful but not market-leading. The result anchor now avoids stale CTA truth, but it still reads more like a good product state than a memorable end-of-session moment.

3. Profile is still more rhythm/status surface than progress identity. It is not broken, but it is the least persuasive commercial surface if viewed outside the first-session proof context.

4. Review repaired/no-repair proof is safe but not final. Open repair readability cleared the bar, yet repaired/no-repair states still have less commercial polish than the core feedback loop.

5. Screenshot proof is usable but scattered. English proof lane exists and recent captures are enough for planning, but a final commercial screenshot pack should wait until the next surface pack lands.

## 6. Runout Comparison

Runout remains stronger on calm packaging, session/result ceremony, visual rhythm, large simple CTAs, and commercial onboarding/paywall readiness.

Sharky is already stronger where it matters for long-term learning trust: honest deterministic repair proof, table-signal feedback, beginner-safe sequencing, same-signal follow-up, and value before monetization.

The principle to borrow from Runout is not its paywall or GTO-heavy posture. The useful principle is consistent premium rhythm: simple screen hierarchy, table or progress as the hero object, calmer completion states, fewer competing labels, and one clear next action.

The next implementation should therefore improve the packaging wrapper around Sharky's existing truth. It should not replace the deterministic proof spine with generic polish.

## 7. Next Implementation Recommendation

Recommend exactly one next implementation wave:

**Product Surface Premium Pack v1**

Intent:

Raise the commercial floor across active first-week surfaces by applying one shared, restrained premium hierarchy to the surfaces that now have correct product truth but uneven presentation.

Recommended bounded scope:

- Home Today/done state: tighten the trainer reason, progress receipt, and primary action hierarchy without changing meaning.
- Practice daily pre-completion/completed state: align the result/progress/CTA structure with Home.
- Review repaired/no-repair state: make fixed proof feel calm and trainer-native, not administrative.
- Profile first-week rhythm: lightly strengthen progress identity if the change is local and does not create a dashboard.
- Learn and Play: guard only. Do not change unless a shared primitive causes an obvious mismatch.

Non-goals:

- No table changes.
- No ModernTable work.
- No route/order/scoring changes.
- No learning content changes.
- No telemetry changes.
- No commerce, paywall, entitlement, or trial work.
- No screenshot tooling work.
- No localization cleanup.

## 8. Why Not Other Candidates

**Feedback Rhythm Pass v1** - Not next. Play/feedback is currently the strongest product proof surface. It can always become calmer, but it is not the highest-EV commercial gap.

**Daily Trainer Entry / Home Session Shell v1** - Not next as an isolated wave. Home and Practice are now good enough; the issue is broader shared surface rhythm, not just Home entry.

**Session Result Ceremony Visual Uplift v1** - Not next as a standalone. Result ceremony is a real future gap, but the completion state is truthful and CTA-safe after the recent corrective.

**Profile Progress Identity v1** - Not next. Profile is below the ideal commercial bar, but it is not the first-session or first-return bottleneck.

**Review Repair Final Polish v1** - Not next. Review open repair cleared the readability bar. Repaired/no-repair polish should be handled as part of the shared pack, not as another Review-only pass.

**Targeted Screenshot Pack Runner v1** - Not next. Capture proof is not the primary blocker. Use screenshots after the next surface pack, not before it.

## 9. Proposed DoD for Next Wave

Product Surface Premium Pack v1 should be accepted only if:

- Home, Practice completed-state, Review repaired/no-repair, and Profile first-week rhythm share a visibly consistent trainer hierarchy.
- Each touched surface keeps one primary user job, one clear reason/progress proof, and one clear next action.
- Existing product truth remains unchanged: no fake AI, no fake rating, no solver/GTO framing, no commercial claims before value.
- Existing deterministic learning signals, repair proof, daily completion state, and English proof lane remain intact.
- Compact phone portrait remains readable without text crowding.
- No table geometry, answer dock geometry, route order, scoring, telemetry, commerce, entitlement, screenshot tooling, or localization behavior changes.
- Focused surface tests, if touched by implementation, prove CTA truth and absence of stale product labels.
- Verification includes formatting for touched files, focused tests for touched surfaces, `./tools/fast_loop_runner_compact_v1.sh` if still the active compact lane, `flutter analyze`, and `git diff --check`.
