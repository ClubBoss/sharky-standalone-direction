# Public Premium TOP1 v1 Endgame Lock v1

## 1. Verdict

public_premium_top1_v1_endgame_lock_ready

## 2. Why this wave exists

Sharky is making real product progress, but the path can still expand into endless valuable-sounding waves: street replay, premium transitions, table text clarity, achievement visuals, Profile proof, value packaging, public readiness, localization, content depth, and external simulator gates.

This lock forces future work to run backward from a bounded final target: Public Premium TOP1 v1 Candidate. Future waves must be selected from the closed route below, must name final-product relevance, and must have measurable DoD. Nice-to-have polish is deferred unless concrete P0/P1 evidence proves it blocks the candidate.

## 3. Current state after Wave 3.0

Closed state:

- proof loop closed: placement / route -> table decision -> feedback -> repair focus -> Practice this spot -> repair outcome -> Session Summary proof -> Review / Profile proof;
- motion / visual proof closed through premium interaction feel and active-shell visual proof;
- Review P1 CTA bridge closed with `Practice this spot`;
- Sharky Soul closed as compact deterministic coach copy, not chat/persona;
- earned reward hook closed as proof-backed Summary payoff;
- Review Pattern Coaching Lite closed as local active-repair focus coaching.

Current state is stronger than narrow beta. It is not yet Public Premium TOP1 v1 because street context, first-week commercial proof, final table text clarity, achievement visual language, profile identity, value packaging, localization boundary, and store/public gates still need controlled closure.

## 4. Final target

Public Premium TOP1 v1 Candidate means a publishable English-first premium poker learning app candidate where the active first-week product feels finished, premium, claim-safe, and competitively differentiated.

It means:

- English-first release surface;
- premium first-week product;
- first proof loop complete and emotionally clear;
- street context understandable;
- Review useful and honest;
- Profile honest;
- rewards proof-backed;
- transitions and motion premium but not noisy;
- runner/table text readable;
- value packaging appears after proof;
- store claims safe;
- final external gate passed.

## 5. Non-requirements for TOP1 v1

Not required for Public Premium TOP1 v1:

- all 36 worlds fully playable;
- full Russian localization;
- full AI coach/chat;
- full hand-history database;
- broad analytics dashboard;
- full badge/RPG economy;
- public solver/GTO trainer;
- leaderboard/community/social systems;
- complete W5-W36 content implementation.

Required stance:

- W1-W4 should be strong, free, and release-visible.
- W5+ should be credible premium/locked/previewed honestly.
- Russian localization is boundary/deferred, not a blocker for English-first TOP1 v1.
- No mixed English/Russian visible copy belongs in English release surfaces.

## 6. Required differentiator: Street Replay / How We Got Here

Street Replay / How We Got Here is required as a TOP1 differentiator because multi-street and turn/river decisions need context. The learner should understand how the current decision arrived here before being asked to choose.

It is not:

- a hand-history archive;
- PokerTracker;
- an all-time hand log;
- an analytics dashboard.

It is:

- a compact learning replay/context scaffold for turn/river or multi-street spots;
- a way to show preflop/flop/turn/river action beats where needed;
- a compact pot, price, and table-context explainer where the current decision depends on prior streets;
- a highlight on the relevant clue;
- support for table-signal learning and causal feedback.

This must be Wave 3.1 unless concrete P0/P1 evidence changes the bottleneck.

## 8. DoD for each future wave

### Wave 3.0.1 - Public Premium TOP1 v1 Endgame Lock

- Goal: freeze the route from current Wave 3.0 state to candidate.
- In-scope: route artifact, TOP1 SSOT pointer, minimal Master Plan/topology trace.
- Out-of-scope: product code, screenshots, Flutter tests, new feature selection.
- DoD: artifact exists, SSOT points to it, no-new-waves rule is explicit.
- Proof required: docs diff, `git diff --check`, `git diff --cached --check`, `graphify hook-check`, status.
- Main drift risk: using the lock as an excuse to add another roadmap family.

### Wave 3.1 - Street Replay / How We Got Here v1

- Goal: make multi-street decisions understandable before the learner chooses.
- In-scope: compact current-hand context scaffold for relevant first-week turn/river or multi-street spots; prior action beats; pot/price/context line; clue highlight; focused tests/screenshots.
- Out-of-scope: full hand-history database, tracker, all-time logs, analytics dashboard, solver/GTO training.
- DoD: a learner can answer "how did we get here?" on the chosen target spot without extra explanation.
- Proof required: focused runner/table tests, screenshot packet for the affected first-week surface, claim-safety review.
- Main drift risk: turning a teaching scaffold into a full hand-history product.

### Wave 3.2 - First-Week Commercial Proof & Gap Lock

- Goal: prove the active first week has enough value and identify only true P0/P1 gaps.
- In-scope: first-week route audit, proof packet, gap classification, blocker/non-blocker calls.
- Out-of-scope: implementing every gap, broad content expansion, public paywall.
- DoD: one first-week packet explains value, trust, proof, and remaining blockers without internal narration.
- Proof required: current first-week/day2 captures, focused docs/review artifact, no generated outputs committed.
- Main drift risk: converting the packet into a broad polish backlog.

### Wave 3.3 - Runner Text Zone & Feedback Clarity Final Pass

- Goal: ensure table/runner copy is readable, spatially clear, and feedback explains the table clue.
- In-scope: text-zone placement, truncation, feedback clarity, minimal copy/layout fixes on active runner surfaces.
- Out-of-scope: Modern Table redesign, new game mechanics, content catalog expansion.
- DoD: active runner screens show readable prompt, clue, answer, and feedback with no overlap or internal vocabulary.
- Proof required: focused runner/widget tests plus compact screenshot packet for touched states.
- Main drift risk: reopening table visual redesign instead of fixing learner-facing clarity.

### Wave 3.4 - Achievement Visual Language / Icons v1

- In-scope: visual grammar for already evidence-backed earned moments; icons or lightweight treatments where proof exists.
- Out-of-scope: broad badge economy, levels, rating, radar, mastery claims, inventory/gallery.

- Proof required: focused earned-moment tests, forbidden-copy scan, screenshots for Summary/Profile if touched.

### Wave 3.5 - Premium Transition & Replay Motion System v1

## 9. No-new-waves rule

No new wave may be inserted into the endgame route unless all are true:

- a P0/P1 blocker is found by tests, screenshots, Claude/Cowork, human tester, or source-truth audit;
- the blocker cannot safely fit into an existing route wave;
- the inserted wave has a bounded DoD and does not expand final target scope.

P2 and nice-to-have items go into deferred notes inside existing waves. They do not create new route waves.

## 10. Anti-theater proof rule

Each future wave must answer:

- what user-visible thing changed?
- which final target requirement moved?
- what evidence proves it?
- what did we explicitly not build?
- why this is not fake progress?

If a wave cannot answer these questions before implementation, it is not ready to run.
