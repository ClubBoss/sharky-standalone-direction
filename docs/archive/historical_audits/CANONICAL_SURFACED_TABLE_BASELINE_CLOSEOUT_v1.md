- compact header budget and protected table viewport are now explicit

- surfaced showdown prompt readability now has a lightweight full-text access path without reintroducing heavy header pressure
- post-push runtime proof on surfaced `w2.s01` after R233 was judged good enough for formal closeout

Closeout boundary:
- no new surfaced table-loop work should start without a serious blocker
- “could look a bit better” is not sufficient grounds to reopen the loop
- old shell remains reference only, not an implementation target

The surfaced baseline now depends on explicit anti-overlap behavior, not local spacing luck.

Canonical expectations:
- surfaced showdown header stays inside a compact fixed budget on portrait-safe surfaced hosts
- surfaced table viewport keeps a protected minimum height
- top text does not silently consume scene-critical space
- showdown top seat and board band keep explicit clearance
- prompt/header text clamps instead of spilling into the scene
- future layout changes must respect protected scene ownership before adding more UI burden

# Reopen Criteria
The surfaced table loop may be reopened only if one of these occurs:
- a reproducible runtime/layout blocker breaks truthful scene reading
- a real regression violates the overlap/viewport contract
- a product-level requirement changes the surfaced host responsibilities

These are not sufficient to reopen the loop:
- minor polish preference
- “one more visual pass”
- desire to chase the old shell more closely without a concrete blocker

# Verification Workflow Rules
Future manual proof for surfaced table work should follow this order:

1. `flutter analyze`
2. one directly affected targeted suite if needed
3. manual runtime proof only when the change is visually meaningful or acceptance-bound

Manual-proof rules:
- prefer a secondary simulator or device for Codex verification when possible
- do not use the user’s primary simulator as the default proof target
- screenshot only when the visual decision actually depends on it
- detach cleanly instead of terminating the app during normal checks
- treat attach/run noise as workflow noise unless there is a reproducible app-side failure
- do not widen into crash work without a real repro path or stack trace

Operational intent:
- keep verification less disruptive
- keep proof loops shorter
- avoid hijacking the user’s main simulator session for routine checks
