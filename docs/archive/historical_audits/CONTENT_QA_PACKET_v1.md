# Content QA Packet v1

Status: ACTIVE
Last updated: 2026-05-06

## Purpose

Provide one short active checklist for approving a bounded content packet
without scattering the decision across multiple docs.

This packet is intentionally small.
Its job is to reduce overthinking, not add a new heavy process.

## Use

Use this before promoting:

1. a bounded slice
2. a session packet
3. a small world packet
4. a transfer/mastery/crucible content packet

Do not use it as a reason to delay obvious good work indefinitely.

## Core Rule

A content packet is ready when:

1. the concept home is clear
2. the learner job is clear
3. the quality bar is clear
4. the main contradiction risk is guarded
5. the residue is named honestly

If those are true, move.
Do not wait for imaginary perfection.

## Five-Gate Checklist

### Gate 1. Route Truth

Confirm:

1. the packet matches `MASTER_PLAN_v3.0.md`
2. disputed concept timing does not violate `CURRICULUM_ROUTE_POLICY_DECISIONS_v1.md`
3. the concept family already has a clear home in `CONCEPT_TO_WORLD_COVERAGE_MATRIX_v1.md`

### Gate 2. Learner Value

Confirm:

1. the packet teaches one real behavior or insight
2. it creates a clearer learner win, not just more material
3. it helps transfer, mastery, repair, or clarity in a visible way

### Gate 3. Content Shape

Confirm:

1. the packet follows the micro-session / atom logic from `CONTENT_SYSTEM_v2.1.md`
2. intro/practice/apply/review shape stays coherent
3. the copy stays concrete, calm, and table-first

### Gate 4. Risk Guard

Confirm:

1. one main contradiction class is guarded or explicitly validated
2. the packet does not rely on hidden runner rescue logic
3. any exclusions are named, not silently tolerated

### Gate 5. Honest Stop

Confirm:

1. remaining residue is bounded
2. the learner-facing path still makes sense without future fill
3. the packet is good enough to improve the product now

If yes, stop and ship the packet.

## Default Decision Order

When reviewing a content packet, use this order:

1. route truth
2. learner value
3. content shape
4. risk guard
5. honest stop

This order prevents both:

1. cosmetic perfectionism before route truth
2. raw expansion without quality control

## Minimal Evidence Rule

The packet does not need every possible proof.

It does need:

1. one clear owner path
2. one clear learner-facing gain
3. one clear guard or validation story

Prefer the smallest proof that catches the real risk.

## Stop Rules Against Overengineering

Stop and avoid further process expansion if:

1. a new checklist item duplicates an existing gate
2. a new doc would only restate this packet with more words
3. the packet is already clearly better and the remaining work is cosmetic
4. new process would slow authoring more than it improves trust

## Relationship To Existing Docs

This packet compresses the active decision path.
It does not replace:

1. `CONTENT_SYSTEM_v2.1.md`
2. `CONTENT_EXCELLENCE_CANON_v1.md`
3. `CONTENT_AUTHORING_CONTRACT_CONTENT_GRAMMAR_v1.md`
4. `CONTENT_FACTORY_RELEASE_GATE_v1.md`

Use those for depth.
Use this packet for go/no-go clarity.

## Practical Outcome

If a packet passes this checklist, the default answer should be:

- promote it
- record bounded residue if any
- move to the next highest-EV packet

Do not reopen the same packet endlessly just because it could still become
slightly prettier.
