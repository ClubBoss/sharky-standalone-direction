# Visual Lift Contract - SSOT (Phase 6.2)

## 1) Purpose
Make the product feel coherent and intentional in day-to-day use by reducing visual friction and inconsistency; this is about product perception and clarity, not decorative beauty.

## 2) Reference dictator
- Single reference: GGPoker.
- No alternative references are allowed during Phase 6.2.

## 3) Visual scope - ALLOWED
- Enforce spacing consistency, alignment, and layout rhythm on existing screens.
- Clarify hierarchy using existing typography and token systems.
- Normalize component states using existing tokens and styles.
- Replace ad-hoc visual overrides with existing token usage where applicable.
- Modern Table: maintenance-only (no redesigns, no restyling).
- No new components; only adjustments to existing ones.

## 4) Visual scope - FORBIDDEN
- New themes, color systems, or type systems.
- New components, new patterns, or new layout paradigms.
- Net-new animations, motion systems, or polish loops.
- "While we are here" changes outside the defined scope.
- Any changes that require new assets or new telemetry.

## 5) Success criteria (DoD)
- Primary flows look like a cohesive product rather than a prototype.
- Visual hierarchy is consistent across entry, session, and completion flows.
- Token usage is consistent and obvious violations are removed.
- No regressions in Modern Table visuals (maintenance-only upheld).

## 6) Timebox and exit rule
- Phase 6.2 is timeboxed (hard stop).
- When the timebox ends, ship the current state even if perfection is not reached.

## 7) Non-goals / drift guards
- Not solving content quality, pedagogy, or lesson structure.
- Not improving system behavior, navigation logic, or telemetry.
- Not expanding personalization, gamification, or monetization.
- Not creating new visual systems or expanding scope beyond existing screens.
